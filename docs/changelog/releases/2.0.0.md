# Release 2.0.0

Release status: `in preparation`

Release date target: `2026-04-09`

This file keeps the implementation history and release notes being prepared for the second major project release.

## Release Summary

### Current preparation scope

- Started the `2.0.0` release track after the completed `1.0.0` launch, keeping release 1 as the latest finalized milestone while documenting the next iteration separately.
- Continued refining the web admin approval experience so release 2 captures polish and usability improvements on top of the already completed core platform flows.

### `current` - Refine pending approval action feedback on the web

- Updated the web `Pending approvals` page so `Approve` no longer makes both action buttons appear to be loading at the same time.
- Reworked the web admin `Overview` pending-preview actions with the same behavior, keeping the row interaction consistent with the full `Pending approvals` page.
- Added a temporary web notification during account approval, matching the existing app feedback pattern already used in request-account flows.
- Kept `Reject` loading scoped to the `Reject` button only, while the opposite action is temporarily disabled to prevent duplicate submissions during processing.
- Updated the temporary-password modal on the web `Pending approvals` page so the `Close` action now uses the same black primary-button styling as the rest of the admin approval flow.
- Removed the placeholder web `Profile` and `Settings` navigation actions for now, replacing them with a black `Logout` button that includes icon + text and asks for confirmation before signing the user out.

### `current` - Polish race management and account navigation on the web

- Added a dedicated web `Personal Options` area with sidebar navigation for `Race types`, `Teams`, `Circuits`, and `Shoes`, including create, edit, delete, discard-confirmation, and in-use dependency handling flows.
- Reworked the in-race managed-option experience so `+` actions inside `Add race` open the same side panel management flow instead of a centered modal, while avoiding unnecessary re-renders in the page behind the drawer.
- Updated bucket-list race actions to show direct `Edit` and `Delete` icons, use anchored confirmation popovers for delete, and return the user to the bucket list after closing race details opened from that view.
- Restored and refined account navigation in the app shell with dedicated `Profile`, `Settings`, and centered `Logout` confirmation flows, plus route-based browser tab titles for the main app areas.
- Built the web `Profile` and `Settings` pages for release 2, including account details, password management, session handling, and a working `Remember me` toggle backed by token storage.
- Improved multiple race-management inputs and forms on the web, including longer race durations, automatic pace calculation, `TimePicker` minute steps, `Analysis notes` sizing, decimal normalization for race-type target km values, and more consistent field-level validation.
- Kept more filters open by default on the web admin `Users` and `Pending Approvals` pages, and aligned `Remember me` and logout styling with the rest of the release 2 design language.

### `current` - Tighten password-change rules and tidy release 2 web settings polish

- Hardened the backend `change-password` flow so a user can no longer reuse the current password as the new password during account updates.
- Added matching frontend validation and a clear inline error message on the web `Settings` page when the entered new password is the same as the current password.
- Simplified the release 2 web `Settings` navigation by removing the temporary `Session` section and keeping the page focused on `Security` and `Preferences`.
- Clarified the web `Remember me` behavior as a browser-local preference: keeping the JWT in `localStorage` when enabled and falling back to `sessionStorage` when disabled.
- Normalized vertical spacing in the web yearly races list so cards inside each year section keep a consistent visual distance from one another regardless of month grouping or card density.

### `current` - Add race follow-up reminders and pending-update shortcuts on the web

- Added a clear post-race `Click to Update` warning on web `Races` cards whenever a race date has already passed but the saved status is still not in a final state.
- Made that warning actionable by opening the edit flow directly from the card badge, while keeping a hover tooltip that explains why the race needs review.
- Added a new header-level pending-update notice on the web `Races` page that surfaces how many races still need a final status review.
- Added a dedicated pending-updates modal that opens from that header notice and reuses the same race cards, details flow, edit shortcut, and actions already available in the main `Races` list.

### `current` - Improve race option management, validation visibility, and empty-state feedback

- Added archive and restore support for personal race options on the web, hiding archived items from the main lists and race selectors by default while allowing them to be revealed through filters.
- Reworked the web `Personal Options` sidebar so its filters match the `Races` visual language and reset when moving between option types, instead of carrying state across sections.
- Improved the web `Add/Edit Race` drawer so tabs with missing required fields show a clear red warning badge when validation fails in another section.
- Fixed the web `Races` filter reset flow so clearing filters while viewing the calendar keeps the user in the calendar instead of forcing a jump back to the list.
- Restored explicit empty-state feedback in the web `Races` calendar views, so production now shows `No races match the current filters.` instead of leaving the content area blank when filters return zero results.
- Fixed a backend race-options SQL issue introduced during archive support that caused race type endpoints to fail for authenticated users, restoring the create/filter/manage option flows.
- Updated the races API documentation examples to remove the retired `wouldRepeatThisRace` field from request and response samples.
